# 软件项目严格验收方法论

本 Skill 规范软件项目的验收报告编写标准，确保每一位验收人产出的报告质量一致、可追溯、可复现。

## 一、核心理念

> **不假设，只验证。不臆想，只测量。**

验收的本质不是"看起来差不多就行"，而是**逐项对照设计文档/需求清单，用数据说话**。每一条 PASS/FAIL 必须有对应的代码证据或运行结果支撑。

---

## 一甲（新增）：验收人准则

验收人不写代码，只做三件事：

1. **逐文件读回** — 独立打开修复工程师声称修改的文件，逐行对照修复声明
2. **实测执行** — `build` / `publish` / `test` / 手动启动，验证运行时行为
3. **证据可视化** — 每一项 PASS/FAIL 必须带有文件路径 + 行号引用

**铁律：永远不信任修复工程师的声明。** 修复组声称"全部通过"、"零遗留"，验收人必须以最坏情况心态独立验证：

| 修复组声明 | 验收人动作 |
|-----------|-----------|
| "pytest N/N" | 自己跑 `pytest -q` |
| "build 0 error" | 自己跑 `build -c Release` |
| "启动正常" | 自己 `publish` → 启动 → 计时 |

> 历史上多次发生"修复组报告 PASS 但验收人实测发现未修"的情况——这恰好证明了独立验收的必要性。

## 二、验收报告标准结构

每份验收报告必须包含以下章节（顺序固定）：

```
1. 报告头 — 任务编号、名称、日期、结论、验收人
2. 验收标准 — 引用设计文档/任务清单原文
3. 详细验收结果 — 逐项对照（表格化）
4. 测试统计 — 总数/通过/失败/通过率
5. 代码质量评价 — 亮点 + 注意事项
6. 问题与改进建议 — 严重度分类 + 修复方案
7. 既有测试评价（如适用）
8. Spec 对照审计
9. 验收结论 — 综合评分 + PASS/FAIL
10. 附录 — 报告生成时间、验收人
```

---

## 三、报告头部规范

```markdown
| 项目 | 内容 |
|------|------|
| **任务编号** | Task X.X / Spec: xxx |
| **任务名称** | 具体名称 |
| **验收日期** | YYYY-MM-DD |
| **测试基准** | N/N |
| **验收结论** | ✅ PASS / ❌ FAIL |
| **验收人** | 验收 Agent |
```

## 四、验收标准章节

从设计文档/任务拆分清单中**原文引用**验收标准，不可凭记忆描述。对于有 `spec.md` 的 Spec 类验收，必须逐条引用 spec 中的 Requirements 和 Scenario。

## 五、详细验收结果 — 逐项对照法

每一项验收点必须用**表格**呈现：

| 序号 | 验收项 | 设计要求 | 实际实现 | 证据 | 状态 |

**必须有证据列**：指向具体文件行号（使用绝对路径格式）。

| 边界值验证 | N/N |

## 七、问题严重度分级

| 等级 | 图标 | 含义 | 处理要求 |
|------|------|------|----------|
| **严重** | 🔴 | 阻塞功能 / 安全漏洞 | **立即修复，修复前不可发布** |
| **中等** | 🟠 | 功能可用但有缺陷 | 近期修复（下一个迭代） |
| **轻微** | 🟡 | 代码质量 / 最佳实践 | 择机优化 |
| **建议** | 🔵 | 增强性改进 / 可选优化 | 参考即可 |

---

## 八、问题记录模板

每个问题必须包含：
- 严重程度
- 问题类型
- 所在文件 + 行号
- 问题描述
- 复现步骤（可执行的复现代码）
- 根因分析
- 修复方案（diff 形式）
- 修复验证结果

---

## 九、修复验证规范

当问题被修复后，在汇总报告中追踪修复状态。修复验证必须包含：
1. 修复前后的具体代码 diff
2. 测试全量回归结果
3. 若无自动测试，需描述手动验证步骤和结果

---

## 十、代码质量评价指南

从以下维度给出评分（⭐1-5）：

| 维度 | 评价项 |
|------|--------|
| 模块分层 | 是否清晰分离 |
| 职责分离 | 是否单一职责 |
| 导出规范 | 公开 API 是否全部导出 |
| 配置管理 | 是否使用最佳实践 |
| 异常处理 | 关键路径是否有降级 + 日志 |
| 类型安全 | 是否有完整的类型声明 |
| 测试覆盖 | 用例数、边界覆盖 |
| 线程安全 | UI 线程 IO 是否隔离 |
| API 兼容性 | 是否有多级 fallback |

## 十三、文件路径引用规范

```markdown
[文件路径#L4-L16](absolute://path/to/file.py#L4-L16)
```

- 必须带行号
- 路径必须为绝对路径

## 十四：Spec 对照审计

如果验收对象是执行 `spec.md` 的 Spec 类任务，必须逐条检查 What Changes 完成度、Scenario 端到端验证、代码 diff 验证。

---

## 十五：退回归档与回归验证

验收通过后必须验证全量回归测试链。旧 PASS 结论不受影响声明必须明确列出。

---

## 十六：置信度审计

当 Spec 涉及重大架构变更时，验收人必须审计可行性等级、最大风险点、架构冗余、工作量估算偏差。

---

## 十七：偏离 Spec 的处理

如果修复方案与 spec 不同但更优，验收人必须：
1. 明确标注"偏离 spec，更优"
2. 写对照表
3. 确认无副作用

---

## 十八：双轨测试体系详解

验收人必须区分两种测试类型：
- **单元测试**：pytest / JUnit / 等框架驱动的单元+集成测试
- **验收脚本**：独立可执行脚本，带 PASS/FAIL 计数

---

## 十九：常见验收陷阱与教训

### 🪤 陷阱 1：相信修复组的 BUILD/PASS 声明
**规则**：永远自己跑 build / test / publish 启动。

### 🪤 陷阱 2：仅在 dev 环境测试
**教训**：`dotnet publish --self-contained` 后的独立可执行文件行为可能完全不同。
**规则**：验收必须使用发布产物，而非开发环境。

### 🪤 陷阱 3：资源路径问题
**教训**：开发环境的相对路径在自包含发布后可能全部失效。
**规则**：所有资源必须用文件系统路径，并做 `File.Exists()` 检查。

### 🪤 陷阱 4："零改动" ≠ 零影响
**规则**：验收人必须理解全链逻辑，而不只看 diff。

### 🪤 陷阱 5：C/C++/C# 等编译语言的静态分析零覆盖
**规则**：手动验证所有编译语言代码路径。`build 0 error` 不等于逻辑正确。

### 🪤 陷阱 6：多模式/多视图切换的完备性（新增）
**实战教训**（天问 25 项 Bug）：7 种模式 + 2 种子模式，`switchMode()` 遗漏恢复逻辑是最高频 Bug（B2/D3/A2/B7 共 4 次）。
**规则**：验收时必须逐个切换所有模式 → 返回原模式 → 验证结果/UI 状态完全恢复。建立"模式切换矩阵"逐格验收。

### 🪤 陷阱 7：跨模式渲染数据不兼容（新增）
**实战教训**（天问 B8）：梅花易数结果调用了五格姓名学的渲染函数 → `strokes` 字段不存在 → TypeError → 白屏。
**规则**：验收时必须测试每种模式组合的渲染输出，验证跨函数调用的数据字段存在性。

### 🪤 陷阱 8：筛选后导航逻辑（新增）
**实战教训**（阴阳先生 第二轮 Bug 3/5）：分类筛选后 prev/next 按钮静默放弃 → 用户点击无效。
**规则**：验收时必须测试"筛选 → 导航 → 边界"全链路，验证按钮禁用/循环回绕逻辑。

### 🪤 陷阱 9：三个环境必须全部验收（新增）
**实战教训**（阴阳先生）：本地开发正常 → dist 构建产物行为不同 → GitHub Pages 线上又不同。
**规则**：验收必须覆盖三个环境：① 本地 dev ② 构建产物（`publish/` 或 `dist/`）③ 线上部署（如有）。每个环境的产物独立验证。

### 🪤 陷阱 10：外部资源降级（新增）
**实战教训**（阴阳先生 Bug 8）：Google Fonts CDN 在中国大陆不可用 → 字体永久阻塞。
**规则**：验收时必须模拟 CDN 不可用场景（Chrome DevTools Network throttling → block 外部域名），验证 fallback 是否生效。

### 🪤 陷阱 11：多策略系统未验证策略"成功"（新增）
**实战教训**（WeChatAuto SmartSendExecutor）：策略执行后不验证目标状态 → 策略"成功"但实际窗口未打开 → 消息发到错误聊天。
**规则**：验收多策略系统时，必须：① 逐个策略执行并验证结果 ② 模拟策略1失败验证降级到策略2 ③ 模拟全部失败验证聚合错误信息 ④ 验证 CLI 自测脚本可独立运行。

## 十九甲：自测循环验收模式（新增）

基于 WeChatAuto 实战：当项目包含自测机制时，验收人需要额外验证自测闭环。

### 验收清单

| 验收项 | 验证方法 | 通过标准 |
|--------|---------|---------|
| CLI 自测脚本独立运行 | `python auto_test.py --target test --text hello` | exit(0) + 输出 OK |
| 自测脚本依赖缺失提示 | 删除依赖后运行 | 输出安装提示，不崩溃 |
| 策略验证逻辑 | 逐个策略执行后检查验证调用 | 每策略都有验证 |
| 策略降级链 | 模拟策略1失败 | 自动切换策略2 |
| 全部失败聚合错误 | 模拟全部策略失败 | 异常包含所有策略错误信息 |
| 成功路径提前返回 | 策略1成功时 | 不执行策略2/3 |

**铁律**：自测机制本身也需要验收。自测脚本通过 ≠ 功能正确，验收人必须独立验证自测脚本的测试逻辑是否覆盖了所有场景。

## 二十、核心检查清单

每次验收前自检：
- [ ] 报告头完整
- [ ] 验收标准原文引用
- [ ] 逐项表格对照（含证据列）
- [ ] 测试统计数据准确
- [ ] 问题有严重度标记
- [ ] 每个问题有复现步骤 + 修复方案
- [ ] 代码质量评价
- [ ] Spec 对照审计完成
- [ ] 全量回归确认
- [ ] 旧 PASS 结论不受影响声明
- [ ] 验收结论明确
- [ ] 所有文件引用使用绝对路径 + 行号
- [ ] 自己跑过 build
- [ ] 自己跑过 publish + 启动
- [ ] 阅读了常见验收陷阱
- [ ] 完成了模式切换矩阵验收（如有多种模式）
- [ ] 验证了跨模式/跨函数调用的数据兼容性
- [ ] 测试了筛选+导航+边界全链路
- [ ] 在三个环境（dev / build产物 / 线上）都跑了验收
- [ ] 模拟了外部资源不可用场景（CDN/字体/API）

## 通用规则（所有 Skill 必须遵守）

1. **任务闭环**：完成任务后必须在 TodoWrite 中将对应任务标记为 ✅ completed，并同步更新管线看板状态。
2. **禁止越权**：严禁逾越自身角色边界。你是验收人，只验证、不开发。超出专业范围的问题应建议切换到对应 Skill。
3. **留痕问责**：在本 Skill 领域内执行了任何任务（完成验收/新增陷阱/更新检查清单等），必须在同目录 `LOG.md` 中追加条目。格式见 Luyi14-project-secretary §4.5。无日志 = 无追溯 = 打回。
